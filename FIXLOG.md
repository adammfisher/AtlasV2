## FX-1 — `users.config.json` invalid JSON silently disables every non-primary account

- **Symptom:** Playwright full suite aborts at collection (`artifacts-bulk-delete.spec.ts:26` SyntaxError); `accounts` parity spec fails (susan/demo/brynn don't exist); any login except the primary account 401s.
- **Evidence:** `node -e "JSON.parse(readFileSync('users.config.json'))"` → `SyntaxError: Expected ',' or ']' after array element in JSON at position 736 (line 7 column 81)`; baseline logs `scratchpad/baseline/playwright.log`, `playwright-parity.log` (accounts spec).
- **Root cause:** a stray token `clauds` was appended after the `brynn` array element (line 7) — an accidental keystroke committed into the working tree during the uncommitted "add brynn" edit. `server/src/lib/account.ts:27-39 accounts()` wraps the parse in try/catch and **silently** degrades to the primary-only FALLBACK list, so the runtime kept working for the primary account and the breakage was invisible until something parsed the file strictly.
- **Why it happened:** hand-edited config with no parse check at edit time, plus a catch-all fallback that hides the failure. (The fallback is correct behavior for resilience; the missing piece is any surfaced signal — noted as a Phase 6 test: broken config must surface a visible warning, and a unit test now locks config validity.)
- **Fix:** removed the stray token. Also added the sanctioned `e2etest` account (isolated DynamoDB partition `A#e2etest|`) used by the new test harness — approved at the Phase 0 gate (open question 3/4).
- **Files changed:** `users.config.json`.
- **Regression lock:** `tests/unit/config.spec.ts` (U-CONF-1) parses `users.config.json` + `models.config.json` + `axiom.config.json` strictly and asserts every account's `models` keys resolve against `models.config.json`.

## FX-2 — any stream close was treated as success; stream lifetime tied to component lifetime (PRIORITY ZERO, part 1)

- **Symptom (user report):** the app returns to the start/home screen before artifact generation completes; artifact creation feels clunky.
- **Evidence:** INFRA-2 at Phase 1 (pre-fix, commit ef52d2f): a transcript cut before its `done` event ended the busy state silently with no error — captured trace in test-results. Baseline parity `x4.spec` (streaming resilience) red; top-level `artifacts.spec` pptx build chain timing out at 4m. Server protocol emits a terminal `done {messageId}` event (`server/src/routes/chat.ts`) that the client never consumed.
- **Root cause:** `client/src/lib/sse.ts` fired `onClose()` on ANY reader close and `ChatView.onClose` unconditionally cleared the live exchange (`setLive(null)`). In a conversation with no persisted messages, `empty = messages.length === 0 && live === null` re-rendered the "What are we building?" home state — mid-generation, whenever the connection dropped (network blip, sleep, proxy restart, CloudFront's 120s origin_read_timeout vs 83–93s silent first-token buffering on big decks). Compounding: the SSE consumer lived inside ChatView, so unmounting (view switch) orphaned the stream; and a mid-stream `read()` rejection left the promise unhandled — permanently stuck busy composer.
- **Why it happened:** the happy path (clean close right after `done`) behaves identically to the failure path at the socket level, so the missing `done` check was invisible in normal use; the `setLive(null)`-always was itself a deliberate patch for an earlier stuck-composer bug (comment preserved in git history) that traded one failure mode for a worse one.
- **Fix:** new `client/src/lib/stream.ts` — a module-level, conversation-keyed stream store. Completion is keyed on the `done` event; close-without-done (and mid-stream read rejection, now caught in `sse.ts`) surfaces an in-place "Connection lost" error with retry; the live exchange is cleared only after the refetched conversation actually contains the persisted `messageId`; user aborts keep the old semantics. ChatView renders store state via `useSyncExternalStore` — the stream survives unmount/nav.
- **Files changed:** `client/src/lib/stream.ts` (new), `client/src/lib/sse.ts`, `client/src/views/Chat/ChatView.tsx`, `client/src/App.tsx`.
- **Regression lock:** A0-1 (×9 kinds), A0-2 (3-minute stream), A0-4a/A0-4b (error + cut), A0-L1/A0-L2 (live, incl. genuine network drop), INFRA-1/2.

## FX-3 — new-chat send never promoted its conversation (PRIORITY ZERO, part 2)

- **Symptom:** sending from the fresh empty state created the conversation server-side, but on stream end the view collapsed back to the empty home state; the chat only existed in the sidebar.
- **Evidence:** paper trace TESTPLAN §3 H5: `send()` created the conversation locally (`target`) but `App.activeConv` stayed null, so ChatView's conversation query stayed disabled and `messages` stayed [].

- **Fix:** `onConvCreated` callback: ChatView reports the created id, App adopts it (`setActiveConv`), the URL updates to `/c/<id>`, and the conv-change effect now skips its live-panel reset when the conversation being switched to is actively streaming.
- **Files changed:** `client/src/views/Chat/ChatView.tsx`, `client/src/App.tsx`.
- **Regression lock:** A0-1 runs on fresh conversations (the exchange must remain on screen with the query active); S-phase adds an explicit empty-state→send→URL test.

## FX-4 — mid-stream error events were erased by the close handler

- **Symptom:** a pipeline error flashed and vanished; the user saw a silent reset instead of the failure.
- **Root cause:** `error` events set `live.error`, but the unconditional `setLive(null)` on close deleted the visible error a frame later.
- **Fix:** in the store, a close after a server `error` event marks the exchange finished but KEEPS it rendered (error box + retry); the composer unlocks via `finished`, so the old stuck-composer failure cannot return.
- **Files changed:** `client/src/lib/stream.ts` (same change set as FX-2).
- **Regression lock:** A0-4a.

## FX-5 — per-chunk synchronous re-render jank ("clunky")

- **Symptom:** the UI stuttered during generation; typing/clicking lagged.
- **Root cause:** every SSE chunk forced a synchronous React state update (hundreds to thousands per generation — site transcript: 1409 gen deltas), each re-parsing the full markdown text and issuing a smooth-scroll `scrollIntoView`.

- **Regression lock:** A0-5 (mid-stream click must respond <1s; long-task counts logged per run).

## FX-6 — network drop during create_doc/edit_doc generation silently discarded the assistant's turn forever (PRIORITY ZERO, live-stack finding)

- **Symptom:** discovered while verifying A0-L2 (the live twin of the mocked A0-4b connection-drop test) against the real stack. A0-L2 failed 3× in a row with the poll for "artifact card or persisted message" never resolving — not a slow-but-eventually-correct result, a permanent one: `GET /conversations/:id` showed only the user's message, no assistant response, indefinitely (confirmed with a direct 5-minute API poll, bypassing the browser entirely).
- **Evidence:** a minimal reproduction script (`fetch` the message POST, read two SSE chunks to confirm the router step fired, `controller.abort()` client-side, then poll the conversation API directly) showed 0 assistant messages after 300s. `pipeline.log` confirmed the router decision logged and nothing else — no `json valid`, no `pipeline error`, nothing — for that conversation, ever.
- **Root cause:** `server/src/routes/chat.ts`'s outer `catch` block is the *only* error handler for the `create_doc`/`edit_doc` pipeline (`runCreateDoc`/`runEditDoc`). It gated all persistence — even an honest "generation failed" message — behind `if (!abort.signal.aborted)`. That guard was written for the plain-chat token-stream's semantics ("a Stop click shouldn't get an ugly error appended"), but that case is handled entirely by a *different*, inner try/catch further up (which already preserves partial prose on user-stop and `return`s before ever reaching the outer catch). `res.on('close')` — the only signal available — cannot distinguish a deliberate Stop from a genuine network drop; both set the same `abort.signal.aborted`. When the client disconnects before `runCreateDoc`'s Bedrock call returns, the call rejects (AbortError), propagates out of `runCreateDoc`, lands in the outer catch, and — because the signal is aborted — every branch that would persist a message was skipped. The user's request was answered by nothing, permanently: no error, no retry affordance, nothing to recover on reload.
- **Why it happened:** the happy path (abort landing *after* the model call already returned, e.g. during the office-file build helper or the chat-summary call) doesn't hit this bug, because later pipeline steps either run in a spawned subprocess with its own independent timeout (unaffected by `ctx.signal`) or catch their own errors internally (`summarize()`) — so the pipeline finishes and persists anyway. Only an early disconnect (before the model call returns) hits the silent-discard path, which is exactly the harder-to-notice, worse-timed case: the more of the (slow, expensive) generation already in flight, the more silently it gets thrown away.
- **Fix:** the outer catch now always persists an honest record — `"Generation was interrupted before it finished (connection lost). Send the request again to retry."` on abort, the existing honest error text otherwise — regardless of `abort.signal.aborted`. Only the *live* SSE push (`sse(res, 'error', ...)`) stays conditional on the client still being connected to receive it; the durable DB write never is. The `OrchestrationError` (missing edit-state) branch got the same treatment for consistency.
- **Files changed:** `server/src/routes/chat.ts`.
- **Regression lock:** `A0-L3` (new, API-level, sub-second — reproduces the exact abort-before-model-returns race deterministically without depending on live generation timing); `A0-L2` (live, end-to-end through the real UI, at real generation speed — re-verified 3× after the fix).
- **Separate test-only defect found and fixed during verification (no product change):** `A0-L2` still failed 3× *after* the product fix above, but the evidence changed shape — trace-level network inspection showed `GET /conversations/:id` returning 200 with real content (1506 bytes, stable) within the first reload cycle every time. The test's reload-poll called `page.locator(...).count()` as an instantaneous snapshot immediately after `composer.waitFor()` resolved — but `page.reload()`'s `load` event fires once the static shell (composer included) mounts, *before* the `useQuery(['conversation', convId])` fetch resolves and re-renders. The count checks were racing that async fetch and reading the DOM before React had anything to show. Fixed in `tests/e2e/live-smoke/a0-live.spec.ts` by giving each reload's content check its own bounded `waitFor` instead of an instantaneous count. Re-verified 3× green (55s-1.6m each, down from hitting the full 400s ceiling every time).

## FX-7 — deterministic router: generic content-nouns (create-md) outscored decisive format-nouns by raw string length

- **Symptom:** found by Phase 3's `AX-multi` test. "Create a two-page onboarding checklist PDF for new analysts, and briefly tell me what you included" routed to `create-md` instead of `create-pdf` — `stage=deterministic conf=1.00`, so not model nondeterminism. `pipeline.log` showed the identical misroute on an unrelated, earlier "onboarding checklist PDF" prompt too (2026-07-17T13:29:32Z) — reproducible, not a one-off.
- **Root cause:** `matchCreate()` (`server/src/pipeline/router.ts`) scores each candidate workflow by `longestHit()` — the character length of its longest matched trigger noun — and picks the unique max. `create-md`'s `nounObjects` list is the deliberate generic catch-all (`doc, notes, guide, plan, outline, checklist, article, blog post, ...`), by design the broadest and last-priority entry in `CREATE_PRIORITY` ("specific → generic", per its own comment). But raw length scoring ignores that priority order entirely: `'checklist'` (9 chars) beat `'pdf'` (3 chars) from `create-pdf`'s much shorter, genuinely unambiguous noun list, even though "PDF" is the one word in the sentence that actually names a file format.
- **Fix (attempt 1, reverted):** first tried excluding `create-md` from the scoring pool whenever any more-specific workflow also had a hit. This over-corrected: re-running the 305-case dataset surfaced a NEW miss ("compose a long-form blog post about remote work" → `create-react-app` instead of `create-md`, because `hasWord()` treats `-` as a word boundary, so `"form"` — from `create-react-app`'s noun list — spuriously matches inside `"long-form"`; harmless under the original scoring since `create-md`'s longer `'blog post'` match always won regardless, but decisive once `create-md` unconditionally stepped aside for ANY competing hit). Phase 3's own `Amd-create` test then caught a second case of the same over-correction live: "Write a README for..." lost to a genuine (non-substring) `"tool"` match from `create-react-app` inside "CLI tool", even though `"readme"` is a far more decisive word for a README request than the generic `"tool"`.
- **Fix (final):** narrower and more correct — a small explicit set of unambiguous file-format words (`pdf, pptx, powerpoint, docx, xlsx, csv, svg, mmd`) is checked FIRST; if exactly one workflow's noun list claims the matched word, that workflow wins outright regardless of any other noun's length. Everything else falls through UNCHANGED to the original longest-matched-noun scoring — so `"readme"` and `"blog post"` keep winning by length exactly as before (the `hasWord()`/`"long-form"` quirk stays latent and harmless, as it always was; not touched).
- **Files changed:** `server/src/pipeline/router.ts`.
- **Regression lock:** re-ran the full 305-case `test:routing` dataset on all 3 tiers post-fix — every metric matches or exceeds the original baseline, zero new misses (small 98.0%/100%/**100%**, mid 99.3%/100%/**100%**, frontier 99.3%/100%/**100%** — unambiguous class actually improved to 100% on every tier, up from 99.5%). New Phase 3 tests `AX-multi` and `Amd-create` lock both the original bug and the over-correction that would have reintroduced a different one.

## FX-8 — office-JSON generation hard-failed on trailing model commentary (unconstrained plain-text path has no defense against it)

- **Symptom:** found by Phase 3's `AX-multi` test, reproduced identically on 2 separate live runs (not a one-off flake). Prompt: "Create a two-page onboarding checklist PDF for new analysts, and briefly tell me what you included." Both times: `Generation failed after repair and salvage: JSON parse failed: Unexpected non-whitespace character after JSON at position 5374 (line 195 column 1)` — the pipeline's own repair (2 attempts) and salvage passes all ran and still failed, because every one of them requires `JSON.parse` to succeed first.
- **Root cause:** office/product JSON generation (`completeJsonOffice` → `bedrockCompleteJson(..., { plain: true })`) always uses `viaPlain()` (`server/src/providers/bedrock.ts`) — raw text streaming, chosen deliberately for smooth live-panel updates, NOT Bedrock's constrained tool-use or json_schema output modes. Nothing structurally stops the model from appending prose after the JSON closes; the system prompt just *asks* for JSON-only. `viaPlain()` only stripped leading/trailing ` ```json ` fences — a trailing sentence with no fence around it (e.g. answering the user's own "...and tell me what you included" verbatim, right after the JSON object) survived straight into `JSON.parse`, which correctly rejects one extra non-whitespace character after a complete, valid value. A request that explicitly invites commentary alongside creation makes this far more likely to trigger — exactly what `AX-multi` exists to probe.
- **Fix:** added `extractJsonValue()` (`server/src/pipeline/validate.ts`) — finds the first balanced top-level JSON object/array in a string (tracking string-literal/escape state so braces inside quoted values don't confuse the depth count) and returns just that span, ignoring anything before or after. `viaPlain()` now runs this after fence-stripping. A genuinely malformed response (no balanced structure at all) still returns the original text unchanged, so `JSON.parse` still reports an honest error rather than this silently masking real breakage.
- **Files changed:** `server/src/pipeline/validate.ts` (new `extractJsonValue`), `server/src/providers/bedrock.ts` (`viaPlain`).
- **Regression lock:** `AX-multi` is the direct reproduction — the same prompt that failed twice now passes 3× consecutively (14.5-15.3s each) after the fix.

## FX-9 — two layered bugs: bundled plugins never provisioned for non-primary accounts, AND every read-modify-write silently no-ops for them anyway (memory was never functional for susan/demo/brynn)

- **Symptom:** found building Phase 4's memory suite. Every test failed the same way: the `remember` tool never fired — the model just answered "Got it, I'll remember that" in plain prose and nothing was persisted. Raw SSE inspection confirmed no `tool` event at all; `route: {intent: "chat"}` with a normal token stream, no `remember-fact` workflow machinery visible in the response.

**Bug 1 — bootstrap never ran for non-primary accounts.**
- **Evidence:** a direct diagnostic (`runAsAccount(user, () => listInstalls())`) showed `adammfisher` (primary) has 7 plugin install rows including `memory` with dozens of accumulated `enabled_projects`; `e2etest` had **zero** install rows — not just memory, *nothing*. Same for `seedIfNeeded()`'s starter projects (p1/p2/p3).
- **Root cause:** `server/src/index.ts` calls `seedIfNeeded()` and `ensureBundledInstalled()` once at server boot, with no account context set. `currentAccount()` (`server/src/lib/account.ts`) falls back to the primary account whenever there's no `AsyncLocalStorage` context — exactly the case at boot, outside any request. So both bootstrap steps only ever ran against the primary account's DynamoDB partition. Every other account in `users.config.json` — **susan, demo, brynn, and e2etest** — starts with zero starter projects and no bundled connector installs.
- **Fix:** `index.ts`'s boot sequence now loops over every account in `accounts()` and runs `seedIfNeeded()` + `ensureBundledInstalled()` inside that account's context via `runAsAccount()`.

**Bug 2 — the deeper one: found because Bug 1's fix alone didn't fix the symptom.** After the boot fix, `e2etest`'s `memory` install row existed (confirmed by diagnostic), but a *freshly created project* still never appeared in its `enabled_projects` — `toolsForProject()` kept returning `[]`.
- **Evidence:** a temporary log line in `projects.ts`'s create handler confirmed `currentAccount() === 'e2etest'` and `mem` was found correctly at request time — the read side was completely fine. Yet the write (`putInstall({ ...mem, enabled_projects: [...] })`) never stuck: re-querying immediately after showed `enabled_projects` unchanged.
- **Root cause:** `server/src/db/appdb.ts`'s `acct(pk)` helper — "one choke point" for account-namespacing every DynamoDB partition key — unconditionally prepends the account prefix: `` `${accountPrefix()}${pk}` ``. Every read-modify-write call site in the file follows the same shape: `put*({ ...rowJustReadFromQueryAll, field: newValue })`. A row returned by `queryAll()`/`getItem()` carries its own **stored, already-prefixed** `pk` field (e.g. `"A#e2etest|PLUGINS"`). That field rides along in the object-literal spread and — because later spread keys win — overwrites the raw `pk: 'PLUGINS'` literal the `put*` helper sets first in the very same object literal. `putItem()` then re-applies `acct()` to this already-prefixed value: `acct("A#e2etest|PLUGINS")` → `"A#e2etest|A#e2etest|PLUGINS"` — a double-prefixed, orphaned key that `listInstalls()` never queries back from (it always asks for the correctly single-prefixed key). The update silently vanishes. **This is a no-op for the primary account**, whose prefix is `''` — `acct(acct(x)) === acct(x)` when the prefix is empty — which is exactly why this has been invisible until a non-primary account exercised a read-modify-write path. It is not specific to the memory install row: it silently breaks *every* update-existing-row call across the whole app (project instructions, conversation titles, artifact versions, settings, anything using this pattern) for susan, demo, and brynn. Memory has never worked for any of them; other update paths may be similarly affected and haven't all been individually audited.
- **Fix:** made `acct()` idempotent — skip re-prefixing when the input already starts with the current account's own prefix (a no-op for primary, where the prefix is empty, matching prior behavior exactly):
  ```ts
  function acct(pk: string): string {
    const prefix = accountPrefix();
    if (!prefix || pk.startsWith(prefix)) return pk;
    return `${prefix}${pk}`;
  }
  ```
- **Files changed:** `server/src/index.ts` (Bug 1), `server/src/db/appdb.ts` (Bug 2, the real fix).
- **Regression lock:** the Phase 4 memory suite (`M-1`..`M-8`) is the direct regression test — all 8 failed identically before either fix (zero tool calls, zero persisted facts), and again after Bug 1's fix alone (bootstrap existed but new-project enablement still silently failed). After both fixes: `M-1`..`M-8` 8/8 green. Full Phase 3 artifact suite (43 tests, e2etest + heavy read-modify-write) re-run clean to confirm no regression from the `acct()` change. Legacy `memory-eval` (primary account) still 14/14 — confirms the fix is a true no-op for primary, as designed.
- **Scope not covered here:** a full audit of every other read-modify-write call site in appdb.ts for latent, now-fixed-at-the-root breakage (the point of fixing `acct()` itself rather than each call site individually) — the choke-point fix covers all of them going forward, but no historical data backfill was performed beyond what `ensureBundledInstalled()`'s own existing-project sweep does automatically on boot.

## Phase 5 harness repairs (test-only, no product change)

Two more instances of the account-auth-predates harness rot pattern already seen in `memory-eval.ts` (FIXLOG's Phase 0 section) and one instance of a harness calling the wrong dispatch function:

- **`test:pipeline-validity`** (`scripts/test/validity.test.ts`): baseline showed `ERROR fetch failed` on every one of 30 prompts — the script never called `ensureBedrockConnected()`/`runAsAccount()`, so `cloudReady()` was false and `completeJson()` fell back to a raw fetch against the retired local-llama server. Fixed by wrapping `main()` in `runAsAccount('adammfisher', ...)` + `ensureBedrockConnected()` (same pattern as `run-routing.ts`). This surfaced a SECOND bug: the script called `completeJson` (the router/classification dispatch — constrained decoding, whatever model happens to be "active") instead of `completeJsonOffice` (what office/product generation actually uses in production — a forced Claude model + deliberately unconstrained plain-text streaming, chosen specifically because complex schemas like `product`'s exceed constrained-decoding grammar limits). Testing the wrong function gave a real-looking but meaningless result: `product-validity` measured 30% (repeated `/scope_in: must be array` violations — a known, documented tool-use type-shape limitation the real pipeline doesn't hit because it never uses that path for this skill). Switching to `completeJsonOffice` brought `product-validity` to 100% and `office-validity` to a stable 90% (gate: ≥90% both) — this is the pipeline's real first-pass validity, not an artifact of testing the wrong code path.
- **`test:stage4-smoke`** (`scripts/test/stage4-smoke.ts`): same auth gap as `memory-eval.ts` predating account middleware — `chat()` sent no Authorization header, every call 401'd instantly (the suspicious "0s" timing on every row), and the unchecked error body was parsed as if it were an SSE stream, silently yielding `tools:[]`/`text:''` instead of a visible failure (exit code stayed 0 throughout). Fixed the auth gap the same way. Doing so revealed the test's own *expectations* are also stale: it expects raw MCP tool names (`memory_upsert`, `graph_add_fact`, `sql_schema`, `sql_query`) that predate a refactor — the real, current, working tool name for remember/forget is `remember`/`forget` (confirmed correct by Phase 4's own passing `M-1`..`M-8` suite), recall now happens via transparent context injection with no explicit tool call visible in the SSE stream at all (also correct — the two "what do you remember..." cases answered the fact correctly despite `called: [none]`), and no `sql_schema`/`sql_query` tool exists post-SQLite-retirement (the model honestly declined both, which is correct behavior, not a bug). Rewriting this script's expectations to match current architecture is real work belonging to Phase 6 (MCP toolloop) — not done here; only the auth-layer fix is applied. One case ("create a file called standup.md...") returned a genuinely empty response (no tool call, no text) rather than an honest decline like the SQL cases — flagged, not investigated further.
- **Files changed:** `scripts/test/validity.test.ts`, `scripts/test/stage4-smoke.ts`.

## FX-10 — live-panel "Retry" resent without clearing the failed attempt, so the model saw its own prior error in context and sometimes refused to comply

- **Symptom:** found re-running the parity suite (`x5.spec.ts`, X5 error recovery). A deliberate Bedrock disconnect → error → reconnect → click Retry sequence intermittently failed: the retried request completed normally (200, real token stream, `done` event) but the model's reply was a refusal — *"This is a system check, and I cannot proceed without a live model connection. Ask a normal question once the connection is restored..."* — even though the connection had genuinely been restored before the retry fired. Reproduced 2 of 6 runs; passed cleanly the other 4, which read as flaky until traced.
- **Evidence:** captured the actual SSE response body for the retried request via the Playwright trace's network resources — the model was mid-conversation with itself: it could see the PRIOR turn's persisted assistant message (*"Something went wrong: No model connected..."*, the honest error FX-6 now always persists) still sitting in the conversation history, and reacted to it as live context rather than a stale artifact of the first failed attempt.
- **Root cause:** two different "retry" code paths exist in `ChatView.tsx`. `retryLast()` (the Regenerate button, for a message already in history) correctly calls `api.truncateConversation(convId, lastUserMsg.id, false)` to drop the failed attempt before resending. The live panel's inline `stream-retry` button (shown while `live.error` is set, mid-conversation) did not — its `onClick` just cleared local client state and called `send(text, true)`, leaving the persisted failed-attempt error message in history for the model to see on the retry. Intermittent because whether the model gets derailed by seeing its own refusal in context, versus just answering the retried request anyway, is itself model-dependent — not a deterministic trigger, which is what made this look like ordinary test flakiness at first.
- **Fix:** the `stream-retry` handler now truncates the failed attempt (same call `retryLast()` already makes, `truncateConversation(convId, lastUserMsg.id, false)`) and invalidates the conversation query before resending.
- **Files changed:** `client/src/views/Chat/ChatView.tsx`.
- **Regression lock:** `x5.spec.ts` — 5/5 clean re-runs after the fix (previously 2/6 failing on identical repeats), and consistently faster (8.5-9.5s vs. 10-90s before, since the model no longer has confusing stale-error context to react to).

## FX-11 — bundled filesystem/sqlite connectors were enabled only for a dead legacy project id, never reachable from any real chat

- **Symptom:** `artifacts.spec.ts`'s "mcp connector tool executes in chat" asked the model to use its filesystem tool; the model replied *"I don't have a filesystem tool available. My toolset includes document reading and creation..."* — the connector was invisible to the model entirely, in the ordinary default ("General") project every sidebar "New chat" lands in.
- **Root cause:** `ensureBundledInstalled()` (`server/src/mcp/manager.ts`) hardcoded `enabled_projects: '["p1"]'` when first installing the `filesystem`/`sqlite` bundled connectors — a leftover from `server/src/db/seed.ts`'s Stage-1 demo fixtures (projects literally id'd `p1`/`p2`/`p3`, with `activeProjectId` defaulting to `p1`). That prototype-era default project was superseded by `conversations.ts`'s `ensureGeneralProject()` (project id `p_general`, "sidebar New Chat must NOT inherit whatever project happens to be active"), but the bundled-connector install code was never updated to match — `toolsForProject(projectId)` gates strictly on `enabled_projects.includes(projectId)`, so `p_general` (and every other real project) never qualified. Only the `memory` connector had special-cased logic to stay enabled for every existing project; `filesystem`/`sqlite` did not, despite the same file's own doc comment describing all of `BUNDLED` as "no credentials, always available." Newly-created projects (via `POST /projects`) had the identical gap — only `memory` got auto-enabled for them.
- **Fix:** `ensureBundledInstalled()` now applies the "enabled for every existing project" treatment uniformly to all of `BUNDLED`, not just `memory`. ## FX-12 — router's file-format decisive-word check was silently non-functional for 6 of its 8 words; "markdown document" misrouted to create-docx

- **Symptom:** `c5-c12-artifacts.spec.ts`'s C9 test — "Create a markdown document: a project readme for ... with a heading, a table of two rows, and a code block." — never produced an `md` artifact (`latest('md', t0)` was `null`). `pipeline.log` showed `stage=deterministic conf=1.00 chosen=create-docx` — a confident, deterministic misroute, not model noise.
- **Root cause:** FX-7's fix added `FORMAT_DECISIVE_WORDS` to `matchCreate()` (`server/src/pipeline/router.ts`) — a small set of format words meant to be checked before the generic longest-noun scoring, exactly to prevent a generic noun like "document" (a `create-docx` trigger) from outscoring a more decisive format word. But the "does this word uniquely belong to one workflow" check looked the word up in each workflow's `nounObjects` trigger array — and of the 8 listed words (`pdf, pptx, powerpoint, docx, xlsx, csv, svg, mmd`), only `pdf` is actually a literal `nounObjects` entry unique to its own create-* workflow. `docx`/`xlsx`/`pptx`/`csv`/`mmd` appear in **no** `CREATE_PRIORITY` workflow's `nounObjects` at all (only in `fileTypeContext`, a different field, not checked); `svg` appears in **two** workflows' `nounObjects` (`create-svg` AND `create-diagram`), so it never resolved to a unique owner either. The mechanism silently no-op'd for 7 of its 8 words the entire time — it happened to look like it worked because FX-7's own regression testing exercised mostly `pdf`-based cases. `markdown`/`md` were never in the list at all, so "markdown document" fell straight through to the generic scorer, where `create-docx`'s `"document"` (8 chars) legitimately outscored `create-md`'s only matching noun, `"readme"` (6 chars).
- **Fix:** replaced the `nounObjects`-lookup with a small hand-authored `word → WorkflowId` map (`pdf, pptx, powerpoint, docx, xlsx, svg, mmd, mermaid, md, markdown`), checked directly rather than derived from trigger tables that were never designed to be a unique format→workflow index. `matchCreate()` now collects every decisive word actually present in the message and only commits if they all agree on one workflow (unchanged safety invariant from FX-7: can't itself introduce a new ambiguity).
- **Regression found by this fix's own verification, then fixed:** the initial version of the new map included `csv → create-xlsx`; the 305-case routing dataset immediately caught a new miss — "build a CLI that converts CSV to JSON" (expects `create-code-artifact`) now misrouted to `create-xlsx`, because unlike the other formats, "csv" is commonly just the *data* a piece of code processes, not the thing being created. `csv` was never functional under the old (broken) mechanism either, so no case relies on it being decisive — removed it from the map rather than adding code-context exceptions.
- **Files changed:** `server/src/pipeline/router.ts`.
- **Regression lock:** full 305-case `test:routing` dataset re-run on all 3 tiers after both the fix and the csv correction — unambiguous 100%/100%/100%, edit-vs-describe 100%/100%/100%, overall 98.4%/99.3%/99.7%, zero new misses vs. the FX-7 baseline. `c5-c12-artifacts.spec.ts` C9 3/3 consecutive, full 8-test file re-run clean.

## FX-13 — the newly-universal filesystem MCP tool (FX-11) caused two new model-confusion regressions: treating project knowledge as filesystem-searchable, and proactively saving long chat replies as files

- **Symptom:** two different failures appeared only after FX-11 made the `filesystem` connector actually reachable everywhere (previously it silently wasn't, so neither had ever surfaced):
  1. `memory-knowledge.spec.ts`'s knowledge-citation test: asked "According to the e2e ops plan document, who is the rollout owner?" with the fact already retrieved into context (confirmed via `/memory/recall-preview`, score 0.71 — well above threshold). The model called `fs_search`/`fs_list` looking for the document as a literal file, found nothing (project knowledge lives in the semantic index, not the sandboxed scratch filesystem), and told the user it had no access — ignoring the answer already sitting in its own context.
  2. `x4.spec.ts`'s "tab close mid-stream" test: asked to "list the numbers from 1 to 500, one per line" (a plain-text streaming prompt, by design, to test the abort-persistence path). The model instead called `fs_write`, saved a `numbers_1_to_500.txt` scratch file, and replied with a one-line "Done" — no long stream ever happened for the abort to interrupt.
- **Root cause:** the `filesystem` connector's tool descriptions (`servers/filesystem.ts`) never distinguished "this project's uploaded Knowledge documents" from "this project's separate sandboxed scratch files" — a reasonable-sounding but incorrect inference once the tool was actually available, worsened by "document"/list-shaped requests reading as file-lookup or file-save opportunities to the model.
- **Fix:** two additions to the per-turn system prompt (`server/src/routes/chat.ts`, alongside the existing `MEMORY:` toolNote, gated the same lightweight way — an early `installFor('filesystem')` check mirroring the existing `memEnabled` pattern, so it rides the cache-stable prefix): a `KNOWLEDGE:` note (fires whenever the project has any knowledge files) clarifying that retrieved passages are already in context and the filesystem tool doesn't reach them; a `FILESYSTEM:` note (fires whenever the filesystem connector is enabled for the project) stating the tool is only for a file the user explicitly names, with a concrete negative example ("list the numbers from 1 to 500...") matching the exact confusion class observed — a plain instruction alone was not sufficient (still reproduced 3/3 on the initial wording; the concrete example fixed it 6/6). Also strengthened `servers/filesystem.ts`'s own `fs_list`/`fs_search` descriptions with the same distinction.
- **Files changed:** `server/src/routes/chat.ts`, `servers/filesystem.ts`.
- **Regression lock:** `x4.spec.ts` 6/6 across 3 repeats; `memory-knowledge.spec.ts` and `m3-m9.spec.ts` (M6, same citation-chip class) each 3/3 consecutive full-file runs; combined re-run of every file touched today (`artifacts.spec.ts`, `memory-knowledge.spec.ts`, `m3-m9.spec.ts`, `x4.spec.ts`, `c5-c12-artifacts.spec.ts` — 24 tests) 2× clean; A0 mocked suite (14 tests) re-confirmed clean.

## Test-only repairs (no product change) — stale selectors and citation-format drift found while chasing FX-11/12/13

Several tests failed for reasons unrelated to any product bug — pre-existing staleness surfaced only once the tests upstream of them started passing far enough to reach the broken assertion:

- **`artifacts.spec.ts` share-button ambiguity:** `button[title*="Share"]` matched two different buttons once V7's conversation-share feature shipped (`ChatView.tsx`'s header "Share conversation" button, added after this test was written) — `.first()` hit the wrong one, so "Share link copied" (the artifact panel's own toast) never appeared. Narrowed to `button[title^="Share: copy"]`.
- **`artifacts.spec.ts` unauthenticated download:** the pptx build-chain test fetched `/artifacts/:id/versions/:v/download` with a bare `fetch()`, no Authorization header — pre-dating the accounts/login gate entirely (the route has required auth ever since, correctly; this is not a case for loosening the API). Added the Bearer token.
- **`memory-knowledge.spec.ts` hardcoded a dead project id:** every endpoint used literal `p1` (`/projects/p1/memory/export`, `/projects/p1/knowledge`) — the same stale legacy-seed id FX-11 found. Sidebar chats live in `p_general`; switched all three tests to it (or, for the knowledge test, to a purpose-created isolated project — see below).
- **Missing project-workspace navigation (`memory-knowledge.spec.ts` ×2, `m3-m9.spec.ts` M4):** all three tests expected a `<button>` containing literal text "Memory" or "Knowledge" reachable directly, without ever opening a specific project's workspace first. Neither exists: "Memory"/"Files" are `<span>` labels in `ProjectWorkspace.tsx`'s inline cards, a sibling of (not a match for) the small pencil-edit `<button>` that actually opens the modal, and that card only renders once a specific project is opened from the Projects list. M4 additionally clicked `button[title*="emory"]` on the bare chat page, which matches a real but unrelated button — the chat header's per-conversation memory on/off toggle, not a modal launcher. Fixed by navigating Projects → the named project card → the modal's own pencil button in all three tests.
- **Page-wide `svg.lucide-x` selector deadlock (`memory-knowledge.spec.ts`):** closing the Memory modal via `page.locator('button:has(svg.lucide-x)').first().click()` matched a per-file "remove" button in the Files card behind the modal's full-screen scrim (same icon, earlier in DOM order) rather than the modal's own close button — Playwright retried the click against a permanently obscured element until the full test timeout (240s) rather than failing fast. Scoped the locator to the modal container.
- **Stale legacy citation-chip selector (`memory-knowledge.spec.ts`, `m3-m9.spec.ts` M6):** both tests checked for `.chat-cite` / `span[title^="From project knowledge"]` — markup `RichText` (`ChatView.tsx`) only emits for the legacy `[source: filename]` prose citation form, used when `recallContext()` is called without a `SourceRegistry`. The actual chat path always passes one, so citations render via the newer index-grounded `<cite index="N-M">` path instead — a numbered `button.chat-chip[data-passage]`, a different class and element entirely. Both switched to the current selector.
- **Shared "General"-project test-data pollution (`memory-knowledge.spec.ts` knowledge test):** with the citation selector fixed, the test still failed intermittently — `p_general` has accumulated documents from every other suite that has ever uploaded through a "New chat" across this project's long test history, and vector search over that much unrelated noise could bury the fixture's one relevant chunk below the recall threshold, or the extra `Filesystem` tool confusion (FX-13) had more surface area to trigger on in a project with a dozen unrelated files already present. Rewrote the test to create and use its own isolated project (`POST /projects` → upload → ask in a conversation scoped to that project id → `DELETE /projects/:id`), with `afterAll` sweeping any `[e2e]`-prefixed project left behind by a failed run.
- **Files changed:** `tests/e2e/artifacts.spec.ts`, `tests/e2e/memory-knowledge.spec.ts`, `tests/e2e/parity/m3-m9.spec.ts`.
