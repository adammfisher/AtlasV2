## FX-1 ‚Äî `users.config.json` invalid JSON silently disables every non-primary account

- **Symptom:** Playwright full suite aborts at collection (`artifacts-bulk-delete.spec.ts:26` SyntaxError); `accounts` parity spec fails (susan/demo/brynn don't exist); any login except the primary account 401s.
- **Evidence:** `node -e "JSON.parse(readFileSync('users.config.json'))"` ‚Üí `SyntaxError: Expected ',' or ']' after array element in JSON at position 736 (line 7 column 81)`; baseline logs `scratchpad/baseline/playwright.log`, `playwright-parity.log` (accounts spec).
- **Root cause:** a stray token `clauds` was appended after the `brynn` array element (line 7) ‚Äî an accidental keystroke committed into the working tree during the uncommitted "add brynn" edit. `server/src/lib/account.ts:27-39 accounts()` wraps the parse in try/catch and **silently** degrades to the primary-only FALLBACK list, so the runtime kept working for the primary account and the breakage was invisible until something parsed the file strictly.
- **Why it happened:** hand-edited config with no parse check at edit time, plus a catch-all fallback that hides the failure. (The fallback is correct behavior for resilience; the missing piece is any surfaced signal ‚Äî noted as a Phase 6 test: broken config must surface a visible warning, and a unit test now locks config validity.)
- **Fix:** removed the stray token. Also added the sanctioned `e2etest` account (isolated DynamoDB partition `A#e2etest|`) used by the new test harness ‚Äî approved at the Phase 0 gate (open question 3/4).
- **Files changed:** `users.config.json`.
- **Regression lock:** `tests/unit/config.spec.ts` (U-CONF-1) parses `users.config.json` + `models.config.json` + `axiom.config.json` strictly and asserts every account's `models` keys resolve against `models.config.json`.

## FX-2 ‚Äî any stream close was treated as success; stream lifetime tied to component lifetime (PRIORITY ZERO, part 1)

- **Symptom (user report):** the app returns to the start/home screen before artifact generation completes; artifact creation feels clunky.
- **Evidence:** INFRA-2 at Phase 1 (pre-fix, commit ef52d2f): a transcript cut before its `done` event ended the busy state silently with no error ‚Äî captured trace in test-results. Baseline parity `x4.spec` (streaming resilience) red; top-level `artifacts.spec` pptx build chain timing out at 4m. Server protocol emits a terminal `done {messageId}` event (`server/src/routes/chat.ts`) that the client never consumed.
- **Root cause:** `client/src/lib/sse.ts` fired `onClose()` on ANY reader close and `ChatView.onClose` unconditionally cleared the live exchange (`setLive(null)`). In a conversation with no persisted messages, `empty = messages.length === 0 && live === null` re-rendered the "What are we building?" home state ‚Äî mid-generation, whenever the connection dropped (network blip, sleep, proxy restart, CloudFront's 120s origin_read_timeout vs 83‚Äì93s silent first-token buffering on big decks). Compounding: the SSE consumer lived inside ChatView, so unmounting (view switch) orphaned the stream; and a mid-stream `read()` rejection left the promise unhandled ‚Äî permanently stuck busy composer.
- **Why it happened:** the happy path (clean close right after `done`) behaves identically to the failure path at the socket level, so the missing `done` check was invisible in normal use; the `setLive(null)`-always was itself a deliberate patch for an earlier stuck-composer bug (comment preserved in git history) that traded one failure mode for a worse one.
- **Fix:** new `client/src/lib/stream.ts` ‚Äî a module-level, conversation-keyed stream store. Completion is keyed on the `done` event; close-without-done (and mid-stream read rejection, now caught in `sse.ts`) surfaces an in-place "Connection lost" error with retry; the live exchange is cleared only after the refetched conversation actually contains the persisted `messageId`; user aborts keep the old semantics. ChatView renders store state via `useSyncExternalStore` ‚Äî the stream survives unmount/nav.
- **Files changed:** `client/src/lib/stream.ts` (new), `client/src/lib/sse.ts`, `client/src/views/Chat/ChatView.tsx`, `client/src/App.tsx`.
- **Regression lock:** A0-1 (√ó9 kinds), A0-2 (3-minute stream), A0-4a/A0-4b (error + cut), A0-L1/A0-L2 (live, incl. genuine network drop), INFRA-1/2.

## FX-4 ‚Äî mid-stream error events were erased by the close handler

## FX-6 ‚Äî network drop during create_doc/edit_doc generation silently discarded the assistant's turn forever (PRIORITY ZERO, live-stack finding)

- **Symptom:** discovered while verifying A0-L2 (the live twin of the mocked A0-4b connection-drop test) against the real stack. A0-L2 failed 3√ó in a row with the poll for "artifact card or persisted message" never resolving ‚Äî not a slow-but-eventually-correct result, a permanent one: `GET /conversations/:id` showed only the user's message, no assistant response, indefinitely (confirmed with a direct 5-minute API poll, bypassing the browser entirely).
- **Evidence:** a minimal reproduction script (`fetch` the message POST, read two SSE chunks to confirm the router step fired, `controller.abort()` client-side, then poll the conversation API directly) showed 0 assistant messages after 300s. `pipeline.log` confirmed the router decision logged and nothing else ‚Äî no `json valid`, no `pipeline error`, nothing ‚Äî for that conversation, ever.
- **Root cause:** `server/src/routes/chat.ts`'s outer `catch` block is the *only* error handler for the `create_doc`/`edit_doc` pipeline (`runCreateDoc`/`runEditDoc`). It gated all persistence ‚Äî even an honest "generation failed" message ‚Äî behind `if (!abort.signal.aborted)`. That guard was written for the plain-chat token-stream's semantics ("a Stop click shouldn't get an ugly error appended"), but that case is handled entirely by a *different*, inner try/catch further up (which already preserves partial prose on user-stop and `return`s before ever reaching the outer catch). `res.on('close')` ‚Äî the only signal available ‚Äî cannot distinguish a deliberate Stop from a genuine network drop; both set the same `abort.signal.aborted`. When the client disconnects before `runCreateDoc`'s Bedrock call returns, the call rejects (AbortError), propagates out of `runCreateDoc`, lands in the outer catch, and ‚Äî because the signal is aborted ‚Äî every branch that would persist a message was skipped. The user's request was answered by nothing, permanently: no error, no retry affordance, nothing to recover on reload.
- **Why it happened:** the happy path (abort landing *after* the model call already returned, e.g. during the office-file build helper or the chat-summary call) doesn't hit this bug, because later pipeline steps either run in a spawned subprocess with its own independent timeout (unaffected by `ctx.signal`) or catch their own errors internally (`summarize()`) ‚Äî so the pipeline finishes and persists anyway. Only an early disconnect (before the model call returns) hits the silent-discard path, which is exactly the harder-to-notice, worse-timed case: the more of the (slow, expensive) generation already in flight, the more silently it gets thrown away.
- **Fix:** the outer catch now always persists an honest record ‚Äî `"Generation was interrupted before it finished (connection lost). Send the request again to retry."` on abort, the existing honest error text otherwise ‚Äî regardless of `abort.signal.aborted`. Only the *live* SSE push (`sse(res, 'error', ...)`) stays conditional on the client still being connected to receive it; the durable DB write never is. The `OrchestrationError` (missing edit-state) branch got the same treatment for consistency.
- **Files changed:** `server/src/routes/chat.ts`.
- **Regression lock:** `A0-L3` (new, API-level, sub-second ‚Äî reproduces the exact abort-before-model-returns race deterministically without depending on live generation timing); `A0-L2` (live, end-to-end through the real UI, at real generation speed ‚Äî re-verified 3√ó after the fix).
- **Separate test-only defect found and fixed during verification (no product change):** `A0-L2` still failed 3√ó *after* the product fix above, but the evidence changed shape ‚Äî trace-level network inspection showed `GET /conversations/:id` returning 200 with real content (1506 bytes, stable) within the first reload cycle every time. The test's reload-poll called `page.locator(...).count()` as an instantaneous snapshot immediately after `composer.waitFor()` resolved ‚Äî but `page.reload()`'s `load` event fires once the static shell (composer included) mounts, *before* the `useQuery(['conversation', convId])` fetch resolves and re-renders. The count checks were racing that async fetch and reading the DOM before React had anything to show. Fixed in `tests/e2e/live-smoke/a0-live.spec.ts` by giving each reload's content check its own bounded `waitFor` instead of an instantaneous count. Re-verified 3√ó green (55s-1.6m each, down from hitting the full 400s ceiling every time).

## FX-7 ‚Äî deterministic router: generic content-nouns (create-md) outscored decisive format-nouns by raw string length

- **Symptom:** found by Phase 3's `AX-multi` test. "Create a two-page onboarding checklist PDF for new analysts, and briefly tell me what you included" routed to `create-md` instead of `create-pdf` ‚Äî `stage=deterministic conf=1.00`, so not model nondeterminism. `pipeline.log` showed the identical misroute on an unrelated, earlier "onboarding checklist PDF" prompt too (2026-07-17T13:29:32Z) ‚Äî reproducible, not a one-off.
- **Root cause:** `matchCreate()` (`server/src/pipeline/router.ts`) scores each candidate workflow by `longestHit()` ‚Äî the character length of its longest matched trigger noun ‚Äî and picks the unique max. `create-md`'s `nounObjects` list is the deliberate generic catch-all (`doc, notes, guide, plan, outline, checklist, article, blog post, ...`), by design the broadest and last-priority entry in `CREATE_PRIORITY` ("specific ‚Üí generic", per its own comment). But raw length scoring ignores that priority order entirely: `'checklist'` (9 chars) beat `'pdf'` (3 chars) from `create-pdf`'s much shorter, genuinely unambiguous noun list, even though "PDF" is the one word in the sentence that actually names a file format.
- **Fix (attempt 1, reverted):** first tried excluding `create-md` from the scoring pool whenever any more-specific workflow also had a hit. This over-corrected: re-running the 305-case dataset surfaced a NEW miss ("compose a long-form blog post about remote work" ‚Üí `create-react-app` instead of `create-md`, because `hasWord()` treats `-` as a word boundary, so `"form"` ‚Äî from `create-react-app`'s noun list ‚Äî spuriously matches inside `"long-form"`; harmless under the original scoring since `create-md`'s longer `'blog post'` match always won regardless, but decisive once `create-md` unconditionally stepped aside for ANY competing hit). Phase 3's own `Amd-create` test then caught a second case of the same over-correction live: "Write a README for..." lost to a genuine (non-substring) `"tool"` match from `create-react-app` inside "CLI tool", even though `"readme"` is a far more decisive word for a README request than the generic `"tool"`.
- **Fix (final):** narrower and more correct ‚Äî a small explicit set of unambiguous file-format words (`pdf, pptx, powerpoint, docx, xlsx, csv, svg, mmd`) is checked FIRST; if exactly one workflow's noun list claims the matched word, that workflow wins outright regardless of any other noun's length. Everything else falls through UNCHANGED to the original longest-matched-noun scoring ‚Äî so `"readme"` and `"blog post"` keep winning by length exactly as before (the `hasWord()`/`"long-form"` quirk stays latent and harmless, as it always was; not touched).
- **Files changed:** `server/src/pipeline/router.ts`.
- **Regression lock:** re-ran the full 305-case `test:routing` dataset on all 3 tiers post-fix ‚Äî every metric matches or exceeds the original baseline, zero new misses (small 98.0%/100%/**100%**, mid 99.3%/100%/**100%**, frontier 99.3%/100%/**100%** ‚Äî unambiguous class actually improved to 100% on every tier, up from 99.5%). New Phase 3 tests `AX-multi` and `Amd-create` lock both the original bug and the over-correction that would have reintroduced a different one.

## FX-8 ‚Äî office-JSON generation hard-failed on trailing model commentary (unconstrained plain-text path has no defense against it)

- **Symptom:** found by Phase 3's `AX-multi` test, reproduced identically on 2 separate live runs (not a one-off flake). Prompt: "Create a two-page onboarding checklist PDF for new analysts, and briefly tell me what you included." Both times: `Generation failed after repair and salvage: JSON parse failed: Unexpected non-whitespace character after JSON at position 5374 (line 195 column 1)` ‚Äî the pipeline's own repair (2 attempts) and salvage passes all ran and still failed, because every one of them requires `JSON.parse` to succeed first.
- **Root cause:** office/product JSON generation (`completeJsonOffice` ‚Üí `bedrockCompleteJson(..., { plain: true })`) always uses `viaPlain()` (`server/src/providers/bedrock.ts`) ‚Äî raw text streaming, chosen deliberately for smooth live-panel updates, NOT Bedrock's constrained tool-use or json_schema output modes. Nothing structurally stops the model from appending prose after the JSON closes; the system prompt just *asks* for JSON-only. `viaPlain()` only stripped leading/trailing ` ```json ` fences ‚Äî a trailing sentence with no fence around it (e.g. answering the user's own "...and tell me what you included" verbatim, right after the JSON object) survived straight into `JSON.parse`, which correctly rejects one extra non-whitespace character after a complete, valid value. A request that explicitly invites commentary alongside creation makes this far more likely to trigger ‚Äî exactly what `AX-multi` exists to probe.
- **Fix:** added `extractJsonValue()` (`server/src/pipeline/validate.ts`) ‚Äî finds the first balanced top-level JSON object/array in a string (tracking string-literal/escape state so braces inside quoted values don't confuse the depth count) and returns just that span, ignoring anything before or after. `viaPlain()` now runs this after fence-stripping. A genuinely malformed response (no balanced structure at all) still returns the original text unchanged, so `JSON.parse` still reports an honest error rather than this silently masking real breakage.
- **Files changed:** `server/src/pipeline/validate.ts` (new `extractJsonValue`), `server/src/providers/bedrock.ts` (`viaPlain`).
- **Regression lock:** `AX-multi` is the direct reproduction ‚Äî the same prompt that failed twice now passes 3√ó consecutively (14.5-15.3s each) after the fix.

## FX-9 ‚Äî two layered bugs: bundled plugins never provisioned for non-primary accounts, AND every read-modify-write silently no-ops for them anyway (memory was never functional for susan/demo/brynn)

- **Symptom:** found building Phase 4's memory suite. Every test failed the same way: the `remember` tool never fired ‚Äî the model just answered "Got it, I'll remember that" in plain prose and nothing was persisted. Raw SSE inspection confirmed no `tool` event at all; `route: {intent: "chat"}` with a normal token stream, no `remember-fact` workflow machinery visible in the response.

**Bug 1 ‚Äî bootstrap never ran for non-primary accounts.**
- **Evidence:** a direct diagnostic (`runAsAccount(user, () => listInstalls())`) showed `adammfisher` (primary) has 7 plugin install rows including `memory` with dozens of accumulated `enabled_projects`; `e2etest` had **zero** install rows ‚Äî not just memory, *nothing*. Same for `seedIfNeeded()`'s starter projects (p1/p2/p3).
- **Root cause:** `server/src/index.ts` calls `seedIfNeeded()` and `ensureBundledInstalled()` once at server boot, with no account context set. `currentAccount()` (`server/src/lib/account.ts`) falls back to the primary account whenever there's no `AsyncLocalStorage` context ‚Äî exactly the case at boot, outside any request. So both bootstrap steps only ever ran against the primary account's DynamoDB partition. Every other account in `users.config.json` ‚Äî **susan, demo, brynn, and e2etest** ‚Äî starts with zero starter projects and no bundled connector installs.
- **Fix:** `index.ts`'s boot sequence now loops over every account in `accounts()` and runs `seedIfNeeded()` + `ensureBundledInstalled()` inside that account's context via `runAsAccount()`.

**Bug 2 ‚Äî the deeper one: found because Bug 1's fix alone didn't fix the symptom.** After the boot fix, `e2etest`'s `memory` install row existed (confirmed by diagnostic), but a *freshly created project* still never appeared in its `enabled_projects` ‚Äî `toolsForProject()` kept returning `[]`.
- **Evidence:** a temporary log line in `projects.ts`'s create handler confirmed `currentAccount() === 'e2etest'` and `mem` was found correctly at request time ‚Äî the read side was completely fine. Yet the write (`putInstall({ ...mem, enabled_projects: [...] })`) never stuck: re-querying immediately after showed `enabled_projects` unchanged.
- **Root cause:** `server/src/db/appdb.ts`'s `acct(pk)` helper ‚Äî "one choke point" for account-namespacing every DynamoDB partition key ‚Äî unconditionally prepends the account prefix: `` `${accountPrefix()}${pk}` ``. Every read-modify-write call site in the file follows the same shape: `put*({ ...rowJustReadFromQueryAll, field: newValue })`. A row returned by `queryAll()`/`getItem()` carries its own **stored, already-prefixed** `pk` field (e.g. `"A#e2etest|PLUGINS"`). That field rides along in the object-literal spread and ‚Äî because later spread keys win ‚Äî overwrites the raw `pk: 'PLUGINS'` literal the `put*` helper sets first in the very same object literal. `putItem()` then re-applies `acct()` to this already-prefixed value: `acct("A#e2etest|PLUGINS")` ‚Üí `"A#e2etest|A#e2etest|PLUGINS"` ‚Äî a double-prefixed, orphaned key that `listInstalls()` never queries back from (it always asks for the correctly single-prefixed key). The update silently vanishes. **This is a no-op for the primary account**, whose prefix is `''` ‚Äî `acct(acct(x)) === acct(x)` when the prefix is empty ‚Äî which is exactly why this has been invisible until a non-primary account exercised a read-modify-write path. It is not specific to the memory install row: it silently breaks *every* update-existing-row call across the whole app (project instructions, conversation titles, artifact versions, settings, anything using this pattern) for susan, demo, and brynn. Memory has never worked for any of them; other update paths may be similarly affected and haven't all been individually audited.
- **Fix:** made `acct()` idempotent ‚Äî skip re-prefixing when the input already starts with the current account's own prefix (a no-op for primary, where the prefix is empty, matching prior behavior exactly):
  ```ts
  function acct(pk: string): string {
    const prefix = accountPrefix();
    if (!prefix || pk.startsWith(prefix)) return pk;
    return `${prefix}${pk}`;
  }
  ```
- **Files changed:** `server/src/index.ts` (Bug 1), `server/src/db/appdb.ts` (Bug 2, the real fix).
- **Regression lock:** the Phase 4 memory suite (`M-1`..`M-8`) is the direct regression test ‚Äî all 8 failed identically before either fix (zero tool calls, zero persisted facts), and again after Bug 1's fix alone (bootstrap existed but new-project enablement still silently failed). After both fixes: `M-1`..`M-8` 8/8 green. Full Phase 3 artifact suite (43 tests, e2etest + heavy read-modify-write) re-run clean to confirm no regression from the `acct()` change. Legacy `memory-eval` (primary account) still 14/14 ‚Äî confirms the fix is a true no-op for primary, as designed.
- **Scope not covered here:** a full audit of every other read-modify-write call site in appdb.ts for latent, now-fixed-at-the-root breakage (the point of fixing `acct()` itself rather than each call site individually) ‚Äî the choke-point fix covers all of them going forward, but no historical data backfill was performed beyond what `ensureBundledInstalled()`'s own existing-project sweep does automatically on boot.

## FX-10 ‚Äî live-panel "Retry" resent without clearing the failed attempt, so the model saw its own prior error in context and sometimes refused to comply

- **Symptom:** found re-running the parity suite (`x5.spec.ts`, X5 error recovery). A deliberate Bedrock disconnect ‚Üí error ‚Üí reconnect ‚Üí click Retry sequence intermittently failed: the retried request completed normally (200, real token stream, `done` event) but the model's reply was a refusal ‚Äî *"This is a system check, and I cannot proceed without a live model connection. Ask a normal question once the connection is restored..."* ‚Äî even though the connection had genuinely been restored before the retry fired. Reproduced 2 of 6 runs; passed cleanly the other 4, which read as flaky until traced.
- **Evidence:** captured the actual SSE response body for the retried request via the Playwright trace's network resources ‚Äî the model was mid-conversation with itself: it could see the PRIOR turn's persisted assistant message (*"Something went wrong: No model connected..."*, the honest error FX-6 now always persists) still sitting in the conversation history, and reacted to it as live context rather than a stale artifact of the first failed attempt.
- **Root cause:** two different "retry" code paths exist in `ChatView.tsx`. `retryLast()` (the Regenerate button, for a message already in history) correctly calls `api.truncateConversation(convId, lastUserMsg.id, false)` to drop the failed attempt before resending. The live panel's inline `stream-retry` button (shown while `live.error` is set, mid-conversation) did not ‚Äî its `onClick` just cleared local client state and called `send(text, true)`, leaving the persisted failed-attempt error message in history for the model to see on the retry. Intermittent because whether the model gets derailed by seeing its own refusal in context, versus just answering the retried request anyway, is itself model-dependent ‚Äî not a deterministic trigger, which is what made this look like ordinary test flakiness at first.
- **Fix:** the `stream-retry` handler now truncates the failed attempt (same call `retryLast()` already makes, `truncateConversation(convId, lastUserMsg.id, false)`) and invalidates the conversation query before resending.
- **Files changed:** `client/src/views/Chat/ChatView.tsx`.
- **Regression lock:** `x5.spec.ts` ‚Äî 5/5 clean re-runs after the fix (previously 2/6 failing on identical repeats), and consistently faster (8.5-9.5s vs. 10-90s before, since the model no longer has confusing stale-error context to react to).

## Flake audit ‚Äî `v7-v12.spec.ts` V10 feedback thumbs: real race, not "just flaky"

## FX-21 ‚Äî the product (PRD) state machine's projections had never been exercised end-to-end: two payload shapes didn't match their own schemas, and the one model-assisted projection had an 8√ó too-small token budget

- **Symptom:** none reported by a user ‚Äî found while writing `product-lifecycle.spec.ts`, the first test ever written to walk the full PRD lifecycle (`server/src/pipeline/product.ts`/`projections.ts`, Amendment 1) past create/edit/version: gated state transitions (proposed‚Üíendorsed‚Üíspecified‚Üíbuilt‚Üíoperating), the six projection kinds, and the bundle export. `artifacts-product.spec.ts` (pre-existing) only ever proved create/edit/version ‚Äî nothing ever called `POST /artifacts/:id/state` or `POST /artifacts/:id/projections` before this session.

  1. **`toConceptDocx`/`toBrdDocx` emitted the wrong payload shape.** They built `{metadata, sections: [{heading, level, paragraphs, table}]}`, but `skills/docx/schema.json` ‚Äî the schema `build_docx.py` actually validates against (the same builder the normal, working docx-creation chat flow uses) ‚Äî requires a FLAT `{metadata, blocks: [{kind: heading|paragraph|table|‚Ä¶}]}`. First real call: `spec validation failed: $: missing required 'blocks' | $: unexpected property 'sections'`.
  2. **`toGatePptx` emitted the wrong shape AND violated a word-count doctrine no JSON schema expresses.** It built `{layout, heading, bullets, col_left, col_right}`; `skills/pptx/schema.json` requires `{archetype, title, speaker_notes, columns: [{head, items}]}` with every slide needing `speaker_notes`. Once the shape was fixed, `scripts/office/validate_common.py`'s `_content_audit` still rejected the output on content: any single bullet/column item over 12 words fails outright, and a "content" archetype slide's title+bullets/columns total must stay ‚â§40 words ‚Äî the original `.slice(0,8)` bullets of full, unbounded sentence length blew both limits.
  3. **`prototype_react` ‚Äî the one model-assisted projection ‚Äî hardcoded `maxTokens: 3072`.** Every other real-document call site in this pipeline uses `officeMaxTokens()` (a 24k budget, model-clamped; per its own comment, "the 8-screen react app measured ~18k output tokens"). 3072 is far below what a real multi-file react app needs; every attempt truncated mid-token: `"the model hit its output ceiling after ~20 characters and the document is cut off mid-token."`
- **Why it happened:** all three transforms were written once, against a payload shape the author had in mind, and never actually round-tripped through the real Python builders or a real model call before this session ‚Äî the same "built but never run" class of gap as `stage4-gates.test.ts` and the M2 script above, just in product code this time instead of test code. Nothing in the existing test suite ever called these code paths.
- **Fix:**
  1. Rewrote `toConceptDocx`/`toBrdDocx` to emit `{metadata, blocks}` directly ‚Äî small `heading()`/`paragraph()`/`table()` block builders replace the old nested-sections shape.
  2. Rewrote `toGatePptx` against the real pptx schema (`archetype`/`title`/`speaker_notes`/`columns`) with a **dynamic word-budget fitter** (`fitToWordBudget`) rather than a guessed fixed item count: each bullet/column-item is word-and-char clipped (`clipWords`, dual-constrained ‚Äî a 12-word clip can still overflow 90 chars on long words), and bullets are greedily kept only while the running total (title words + item words) stays ‚â§40, so it adapts to however much of the PRD's real content fits rather than silently dropping or overflowing.
  3. Changed `prototype_react`'s `completeJson` call from `{maxTokens: 3072}` to `{maxTokens: officeMaxTokens()}`.
- **Files changed:** `server/src/pipeline/projections.ts`.
- **Regression lock:** `tests/e2e/live-smoke/product-lifecycle.spec.ts` 16/16 √ó 3 consecutive; `tests/e2e/live-smoke/artifacts-product.spec.ts` (pre-existing create/edit/version coverage) re-verified still green; full 305-case routing dataset unaffected (this fix touched no router code).

## FX-22 ‚Äî "markdown one-pager PRD" fell through the router's markdown-deliverable regex and misrouted into editing an unrelated prior artifact

- **Symptom:** found writing `project-owner-prototypes.spec.ts` (a product owner's real session: JSX prototype, then a markdown PRD one-pager, then a docx brief, all in one project). The second message ‚Äî "Create a markdown one-pager PRD for the loan payment calculator: problem, target users, and 3 success metrics" ‚Äî never produced an `md` artifact.
- **Evidence:** `pipeline.log` showed `route ... chosen=edit-code-artifact ... stage=deterministic conf=1.00` for that exact message, immediately after the react prototype's own creation ‚Äî the router treated a request for a NEW markdown document as an edit of the react app that had just been created in the same conversation.
- **Root cause:** `MARKDOWN_DELIVERABLE_RE` (added in FX-15 to stop bare "markdown"/"md" from misfiring on inline-formatting mentions like "a markdown table") only recognized `markdown (document|doc|file|artifact|readme)`, `(as|in|to) markdown`, or a literal `.md`. Because a react artifact already existed in this conversation (`signals.artifactInContext`, `lastArtifactKind: 'react'`), Stage 1's step 8 ("a real produce-verb naming the existing artifact's OWN type ‚Äî e.g. 'make the deck darker'") saw `hasCreate.skill === targetKind` and confidently rerouted the whole request to `edit-code-artifact`.

## FX-23 ‚Äî every chat reply, on all three model providers, silently truncated at 2048 output tokens with no error shown (PRIORITY ZERO)

- **Symptom (user report):** asked a project "tell me what you know about this project" ‚Äî a real reply that recites a large amount of recalled context. The reply just stopped mid-sentence: `"...Explored **signage SaaS** as"` ‚Äî no error box, no retry affordance, no indication anything was wrong. The user only noticed because the sentence obviously didn't finish.
- **Evidence:** `pipeline.log` for that exact turn: `usage conv=c_mrtfga99e0l5uv in=10708 out=2048` ‚Äî an exactly-round `2048`, the unmistakable signature of hitting a hardcoded cap rather than a model's own (non-round) output ceiling. The stored message text in DynamoDB confirmed it: 8977 characters, ending literally at `"...Explored **signage SaaS** as"` with nothing after.
- **Root cause:** `chat.ts`'s only entry point into inference for a normal conversational reply is `streamWithTools()` (`providers/dispatch.ts`), and it has never once passed an explicit `maxTokens` ‚Äî not by oversight in one call, but because the option was never threaded through at all. Every provider's `streamWithTools` implementation therefore fell back to its own internal default when `opts.maxTokens` was undefined, and all three had the same flaw: a **flat, arbitrary constant** instead of the model's real ceiling (`modelMaxOutput()`, which already existed and is used correctly elsewhere in this exact file for document generation):
  - `bedrockStreamWithTools` (bedrock.ts): `?? 2048` ‚Äî for Nova 2 Lite (`maxOutput: 8192`), a 4x-too-small cap; for Haiku (64000) or Sonnet (128000), 30-60x too small.
  - `openai.ts` / `anthropic.ts` `streamWithTools`: `?? 4096` ‚Äî same bug, same shape, reached through the exact same `chat.ts:450` call site whenever an OpenAI or Anthropic model is the active model.
  - Two sibling functions had the identical `?? 2048` pattern (`bedrockCompleteText`, and `completeText` in both adapters) ‚Äî currently only reached by callers that either pass their own explicit budget or genuinely want a short output (title/descriptor generation), so not actively truncating anything today, but the same landmine for the next caller that doesn't.
  - `bedrockStreamMessages`/`bedrockStreamChat` had it too, but are dead code (zero callers anywhere in the repo, confirmed by full-repo grep) ‚Äî fixed anyway since leaving a known-bad default in reachable-looking code is how the next regression gets written.
- **Why it happened:** this is the exact same bug shape as FX-21's `prototype_react` (`maxTokens: 3072` instead of `officeMaxTokens()`) and FX-2's original stream-handling gap, but in the single most-travelled code path in the entire app ‚Äî every ordinary chat turn, on every provider ‚Äî which is precisely why nobody had to write a special prompt to hit it: any reply that runs even moderately long (reciting recalled memory, summarizing a document, walking through a multi-part answer) crosses 2048 tokens (~1500 words) constantly. The 2048/4096 constants look like reasonable "safe default" numbers in isolation, which is exactly what made them easy to leave unquestioned at every one of these call sites rather than wiring through `modelMaxOutput()` like the document-generation paths already do. And because a `max_tokens` stop is a completely normal, non-error `stopReason` from the model's perspective, nothing downstream had any signal to distinguish "the model finished" from "the model was cut off" ‚Äî the stream just ends, `chat.ts` emits its normal `done` event, and the truncated text gets saved and rendered exactly like a complete reply.
- **Fix:** every one of the six call sites above now defaults to `opts.maxTokens ?? modelMaxOutput(...)` (using the correct model for context ‚Äî `modelDefByModelId(modelId)` inside the tool loop, since it resolves per-call, not just the process-wide active model). No caller anywhere in the repo relies on the small constants on purpose (audited via full-repo grep across every call site of all six functions before changing anything) ‚Äî the ones that need a short output already pass their own explicit `maxTokens` (200-700 range), and none of them regress.
- **Files changed:** `server/src/providers/bedrock.ts` (`bedrockStreamWithTools`, `bedrockStreamMessages`, `bedrockCompleteText`), `server/src/providers/openai.ts` (`streamWithTools`, `completeText`), `server/src/providers/anthropic.ts` (`streamWithTools`, `completeText`).
- **Regression lock:** reproduced the exact failure (fresh conversation, same "tell me everything about this project" style prompt against the same project) ‚Äî before the fix, confirmed truncation; after, `usage ... out=2603` (past the old 2048 ceiling) with a clean `done` event and the stored message ending on a complete sentence. `tests/e2e/chat.spec.ts`, `tests/e2e/memory-knowledge.spec.ts`, and `tests/e2e/parity/x-polish.spec.ts` (X1, which specifically compares reply lengths) re-run to confirm no regression in ordinary-length replies, tool-use, or stop/regenerate behavior.
